============================================================
 VIRTUAL SCROLL — INTENTO FALLIDO (2026-03-15)
 Investigación, código aplicado, y razones del fracaso
============================================================

OBJETIVO
--------
Con 300+ archivos la tabla de resultados iba lenta al hacer scroll.
Causa raíz: cada fila = ~15 widgets CTk (1 CTkFrame + 7 cell frames +
7 labels + QualityBadge). Con 300 archivos = ~4500 widgets en un
CTkScrollableFrame. Tkinter reposiciona TODOS en cada scroll event.

La idea: reemplazar CTkScrollableFrame por un tk.Canvas + CTkScrollbar
manual, con un pool de ~25 ResultRow reutilizables. Solo renderizar
las filas visibles, reciclando widgets al scrollear.


ARQUITECTURA IMPLEMENTADA
-------------------------

Antes:
  CTkScrollableFrame contiene 300 ResultRow widgets (todos pack'd)

Después:
  tk.Canvas + CTkScrollbar + tk.Frame (inner_frame)
  _data_list: List[AnalysisResult]  → datos puros, orden de display
  _data_map: Dict[str, AnalysisResult]  → lookup O(1) por filepath
  _selected_keys: Set[str]  → selección basada en filepaths
  _pool_free: List[ResultRow]  → widgets libres para reciclar
  _row_map: Dict[int, ResultRow]  → índice visible → widget asignado
  _render_visible()  → corazón: recicla widgets según scroll position

POR QUÉ FALLÓ (rendimiento peor que antes)
-------------------------------------------
Los tests (82/82) pasaron, el boot y análisis funcionaban, pero el
rendimiento visual fue PEOR que la implementación original.

Hipótesis de por qué:

1. PROBLEMA CON tk.Canvas + CTk widgets:
   customtkinter widgets (CTkFrame, CTkLabel) tienen lógica de rendering
   interna pesada (doble canvas, esquinas redondeadas, scaling). Montar
   estos widgets dentro de un tk.Canvas crudo via create_window + place()
   puede ser más lento que el CTkScrollableFrame optimizado internamente
   por customtkinter. El CTkScrollableFrame ya usa internamente un canvas
   pero con optimizaciones específicas para CTk widgets.

2. _render_visible() SE LLAMA DEMASIADO:
   Cada mousewheel event → yview_scroll → _render_visible(). En macOS
   con trackpad, se generan decenas de eventos por gesto. Cada llamada
   hace: winfo_height/width, yview(), set operations, y potencialmente
   update_result() + place() en varios widgets. No hay throttling ni
   debouncing en _render_visible().

3. update_result() ES CARO CON TRANSICIONES:
   Cuando se recicla un widget de "Pendiente" a "Analizado" (o vice
   versa), update_result() destruye el label/badge antiguo y crea uno
   nuevo. Esto incluye crear CTkFrame, CTkLabel, bind events recursivo.
   Con pool recycling esto pasa frecuentemente al scrollear.

4. place() + place_forget() OVERHEAD:
   CTkFrame.place() y place_forget() en customtkinter no son tan
   ligeros como en tk puro. Cada llamada puede triggear re-render
   interno del widget CTk.

5. _get_visible_data() CREA LISTAS NUEVAS:
   Cuando hay filtro activo, cada _render_visible() crea una nueva
   lista filtrada. Sin filtro, devuelve referencia directa (OK).

6. MOUSEWHEEL BINDING GLOBAL:
   bind en toplevel con filtrado por posición añade overhead a TODOS
   los mousewheel events en la app, no solo los del canvas.

7. FALTA DE yscrollincrement:
   Sin configurar yscrollincrement, yview_scroll(-delta, "units")
   usa el default de Tk (1/10 del viewport), lo cual puede resultar
   en scroll demasiado rápido o con saltos visibles.


LECCIONES APRENDIDAS
--------------------
- customtkinter widgets NO son equivalentes a tk widgets estándar
  para operaciones de pool/reciclaje. Su overhead interno (doble
  canvas, CTkBaseClass, scaling) hace que operaciones como place(),
  place_forget(), y configure() sean significativamente más lentas.

- El approach correcto probablemente requiere:
  a) Usar ttk.Treeview (widget nativo) en vez de CTk widgets custom,
     O
  b) Mantener CTkScrollableFrame pero implementar lazy loading
     (crear widgets bajo demanda cuando se acercan al viewport,
     destruir cuando se alejan mucho), con debouncing agresivo,
     O
  c) Virtualizar usando tk.Canvas con canvas items nativos (text,
     rectangle) en vez de widgets embebidos — los canvas items son
     mucho más ligeros que widgets.

- Los tests de CI (UI tests) NO detectan problemas de rendimiento.
  Solo verifican que la app arranca y que los métodos no crashean.
  Se necesita testing manual con 300+ archivos para evaluar scroll.

- La complejidad del cambio fue alta (reescritura de ~600 líneas)
  para un resultado negativo. Un approach más incremental (ej:
  solo añadir debouncing al scroll existente) habría sido más seguro.
